# CSR分析技能 — IATF 16949 客户特定要求判断

## 概述

分析用户提供的文档（PDF/Word/Excel/TXT），判断其中哪些条款属于 IATF 16949 汽车质量管理体系下的客户特定要求，逐条输出结构化判断报告，附带置信度评分。

**能力边界（本技能不做以下事情）：**
- ❌ 不提供 IATF 16949 标准全文解读（仅判断 CSR 归属性）
- ❌ 不提供合规咨询或整改建议（仅做分类判断）
- ❌ 不替代第二方审核或客户官方 CSR 确认
- ❌ 不处理非汽车行业的质量管理体系文档（如 ISO 9001 通用要求、医疗器械 ISO 13485 等）
- ❌ 不从互联网检索 OEM CSR 文档（仅分析用户提供的文档）

## 触发检测

当用户消息匹配以下任一模式时激活本技能：

**直接触发（精确匹配或近似匹配）：**
- `判断CSR` / `CSR审核` / `CSR分析` / `IATF CSR`
- `客户特定要求` / `客户特殊要求` / `顾客特定要求` / `顾客特殊要求`

**语义触发（组合模式）：**
- "客户要求" + "质量体系" / "16949"
- "OEM特殊要求" / "OEM额外要求"
- "汽车客户" + "要求" / "手册" / "标准"
- "供应商质量手册" / "供应商要求手册"

**文档分析触发（带附件）：**
- `分析这份文档` / `帮我看看这份` / `检查这个文件`（当用户上传附件时）
- 任何带有汽车/质量相关上下文文件上传

**负面触发（以下情况不应激活本技能）：**
- 用户仅询问 IATF 16949 标准条款含义 → 用通用知识回答
- 用户要求编写质量管理体系文件 → 不属于 CSR 判断范畴
- 用户上传的文档明确与汽车行业无关（如食品、医疗、建筑） → 不激活

## 输入规范

| 属性 | 要求 |
|------|------|
| 支持格式 | PDF、DOCX、XLSX、TXT |
| 文件大小 | 单文件 ≤ 50MB；超出时提示用户拆分 |
| 语言 | 中文、英文、中英混合 |
| 文档类型 | OEM CSR 文件、供应商质量手册、质量协议、客户要求清单等 |
| 附件数量 | 单次分析 1 个文件；多文件时逐个处理 |

## 工作流程

### 步骤一：接收并解析文档

- 接受 PDF、DOCX、XLSX 或 TXT 格式文件。
- 使用对应工具提取全文（PDF 用 pdf 技能、Word 用 docx 技能、Excel 用 xlsx 技能）。
- 对于扫描型 PDF（图片版），先尝试 OCR 提取。若文本无法提取，向用户报告并要求提供可读版本。
- 提取过程中保留所有条款/章节编号和层级结构。

**异常处理：**
- **空文档或无可提取文本**：报告"文档内容为空或无法提取文本"，停止分析，建议用户检查文件。
- **超大文档（>50MB）**：提示用户拆分后重新上传，不强行处理。
- **不支持的格式**（如图片、压缩包）：明确告知不支持，列出支持的格式。
- **文本提取不完整**（如 PDF 部分页面为扫描件）：标注提取覆盖率，仅对已提取部分分析，未提取部分在报告中标注。

### 步骤二：预筛查

在详细分析之前，先进行快速预筛查：

1. **文档是否来自已知汽车 OEM 或一级供应商？**
   检查公司标识（logo、页眉、页脚、文档属性）。
   参考 `references/csr-knowledge-base.md` 中的已知 OEM 列表。

2. **文档是否引用了 IATF 16949 或 ISO/TS 16949？**
   搜索关键词："IATF 16949"、"ISO/TS 16949"、"16949"、"质量管理体系"。

3. **文档是否提出了超出 IATF 16949 标准的要求？**
   寻找语言模式："除……之外"、"还应"、"也必须"、"超过"、"更严格"、"至少"、"不低于"。

如果三个预筛查问题均为否定，文档极不可能包含 CSR——直接说明结论并跳至总结。

**参考文件缺失处理：** 如果 `references/csr-knowledge-base.md` 不存在或无法读取，不中断分析，改用内置的 IATF 16949 常见 CSR 条款知识进行判断，并在报告中注明"参考知识库未加载，基于通用知识判断"。

### 步骤三：逐条分析

对文档中的每条要求条款，按以下四个维度逐一分析：

| 维度 | 判断问题 | 判断指标 |
|------|----------|----------|
| **维度一：IATF 16949 引用** | 该条款是否明确引用了 IATF 16949 条款编号？ | "根据 IATF 16949 §X.X.X"、"除 §X.X.X 条款外"、交叉引用表 |
| **维度二：汽车 OEM 来源** | 该要求是否由汽车 OEM 或一级供应商发布？ | 文档中的公司名称、OEM 专有格式、供应商门户来源 |
| **维度三：超出标准要求** | 该条款是否提出了超出 IATF 16949 标准的要求？ | "还应"、"另外"、"超过"、"比……更频繁"、"比……更严格"、"至少" |
| **维度四：强制性** | 该要求是强制性的（必须/应）还是建议性的（宜/可）？ | "必须"、"应"、"要求" vs. "宜"、"建议"、"可" |

### 步骤四：分类与置信度评分

对每条条款，赋予三种标签之一并附带置信度评分（1-5 分）：

- **属于 CSR** — 至少维度一+维度三或维度二+维度三为正面，且维度四为强制性。
  - 置信度 5：四个维度全部满足，依据明确
  - 置信度 4：三个维度满足，依据较明确

- **存疑** — 维度部分匹配，信息不足以确认。
  - 置信度 3：两个维度满足，但缺少关键信息
  - 置信度 2：仅一个维度满足，信息不足

- **不属于 CSR** — 不满足 CSR 分类标准。
  - 置信度 5：明确无 IATF 16949 关联且非汽车客户
  - 置信度 4：标准原文复述，无额外要求

### 步骤五：质量自检

在生成报告前，执行以下自检：

1. **覆盖率检查**：文档中所有疑似要求条款是否均已分析？如有遗漏，补充分析。
2. **一致性检查**：相似条款的判断结果是否一致？如有矛盾，复查并调整。
3. **依据完整性**：每条判断是否都有一句话依据？无依据的条目标记为"存疑"。
4. **边界情况复核**：参考 `references/csr-knowledge-base.md` 中的"判断中常见的边界情况"，确保已正确处理。

### 步骤六：生成报告

同时生成完整的 Markdown 报告文件和在对话中输出摘要。

**6a. 将完整 Markdown 报告写入文件：**

在当前工作目录创建名为 `CSR判断报告-{文档简称}.md` 的文件。文件必须包含完整分析，涵盖以下章节。不要截断或摘要——每条分析的条款都必须包含在内。

报告文件结构：

```markdown
# CSR 判断报告

## 文档信息
- **文件名**：
- **文档类型**：
- **文档来源（OEM/供应商）**：
- **涉及 IATF 16949 版本**：
- **分析日期**：

## 预筛查结果

| 筛查维度 | 结果 | 依据 |
|----------|------|------|
| 汽车行业客户来源 | ✓/✗ | 具体依据 |
| IATF 16949 引用 | ✓/✗ | 具体依据 |
| 超出标准要求 | ✓/✗ | 具体依据 |

## 逐条判断

| 序号 | 条款编号 | 条款摘要 | 维度一 | 维度二 | 维度三 | 维度四 | 判断结果 | 置信度 | 判断依据 |
|------|----------|----------|--------|--------|--------|--------|----------|--------|----------|
| 1 | | | ✓/✗ | ✓/✗ | ✓/✗ | 强制/建议 | 属于CSR/存疑/不属于CSR | 1-5 | 一句话依据 |

## 汇总统计

- **属于 CSR 条目**：N 条（其中置信度≥4：N条）
- **存疑条目**：N 条
- **非 CSR 条目**：N 条
- **平均置信度**：X.X / 5.0

### IATF 16949 章节分布

| 章节 | CSR 条目数 | 涉及条款 |
|------|:---------:|----------|
| §X.X | N | 条款摘要 |

### 关键差异点对比

| 维度 | IATF 16949 标准要求 | 本协议 CSR 要求 |
|------|---------------------|-----------------|
| | | |

## 质量自检结果

| 检查项 | 结果 |
|--------|------|
| 覆盖率（所有疑似条款已分析） | ✓/✗ |
| 一致性（相似条款判断一致） | ✓/✗ |
| 依据完整性（每条有依据） | ✓/✗ |
| 边界情况已复核 | ✓/✗ |

## 结论

（一段总结性文字，描述文档的 CSR 特征和关键发现，包含整体置信度评价）
```

**6b. 在对话中输出简要摘要：**

写入文件后，向用户展示简洁摘要：分析条款总数、CSR 条目数、存疑条目数、非 CSR 条目数、平均置信度，以及报告文件路径。使用 `present_files` 将报告文件呈现给用户。

**关键规则**：绝不跳过文件写入步骤。`.md` 报告文件是主要交付物——对话中的摘要是次要的。

## 约束条件

1. **始终将完整报告写入 `.md` 文件。** 报告文件是主要交付物。不要仅在对话中输出摘要——完整分析必须持久化到磁盘。使用 `present_files` 向用户呈现。
2. **不要解释 CSR 定义或 IATF 16949 背景**，除非用户明确要求。只执行判断。
3. **默认输出中文**（用户为中文使用者）。保持格式清晰、结构规整。
4. **对于扫描/图片 PDF 无可提取文本的情况**：尝试基础 OCR；若失败，报告局限性，不虚构分析结果。
5. **存疑时**：使用"存疑"而非强制二分分类。用户可提供额外上下文。
6. **如果文档与汽车质量完全无关**：明确说明并停止——不要强行匹配条款。
7. **不要访问互联网**查找 OEM CSR 文档——仅分析用户提供的内容。
8. **不要存储或传输文档内容**到分析会话之外——不缓存、不记录到外部服务。

## 信任与安全声明

- **数据处理**：用户上传的文档仅在当前会话中分析，分析完成后不保留文档内容。报告文件仅包含分析结论，不包含原文完整复制。
- **权限边界**：本技能仅读取用户主动提供的文件，不主动访问文件系统其他位置，不执行网络请求。
- **安全红线**：本技能不执行任何脚本代码，不调用系统命令，不修改用户文件（仅创建分析报告文件）。
- **中文优先**：所有输出默认使用中文，符合国内用户使用习惯。

## 分析示例

以下是一个完整的判断示例，展示本技能的输出标准：

> **输入条款**："根据 IATF 16949 第 8.3.4.4 条，供应商必须在产品批准过程中提交 Level 4 PPAP，且过程能力指数 Cpk 必须 ≥ 1.67（标准要求 Cpk ≥ 1.33）。"

| 维度 | 结果 | 依据 |
|------|------|------|
| 维度一：IATF 16949 引用 | ✓ | 明确引用 §8.3.4.4 |
| 维度二：OEM 来源 | ✓ | OEM 要求 |
| 维度三：超出标准 | ✓ | Cpk≥1.67 vs 标准 1.33 |
| 维度四：强制性 | ✓ | "必须" |
| **判断** | **属于 CSR** | 置信度 5/5 |

更多示例见 `references/csr-knowledge-base.md` 中的"CSR 判断示例"章节。
